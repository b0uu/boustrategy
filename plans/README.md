## Human checkpoints — where autonomy must stop

Plans 001-008 are executable with zero maintainer input (008 ends in a
report). Past them, these decisions belong to the maintainer and no agent
should improvise around them:

1. **X provider + budget** — decided from plan 008's report. Gates the X
   ingestion build.
2. **Curated X account list** — human-only curation per
   `docs/source_policy.md`. Gates any real X signal generation.
3. **Theme memos** (14 themes, spec §4) — agents may draft, the maintainer's
   bull/bear cases and invalidation instincts make them real. Gates
   high-quality thesis chains.
4. **Approved/rejected historical trade examples** — preference-alignment
   eval data only the maintainer can label.
5. **Reasoning prompts sign-off** — thesis-chain and daily-management
   prompts can be drafted from the docs, but the maintainer approves before
   any decision-generating run.
6. **Anything touching live money** — Robinhood account, credentials,
   broker adapter activation (Phase 2). Never autonomous.

## Maintainer decisions (resolved)

2026-07-08, working session:

- RED/DE_RISKING gate: **escalation gate** — BUY/ADD rejected unless the
  record declares `extraordinary_opportunity` with justification. Full
  normal sizing caps apply to extraordinary trades in all regimes.
- Trade quota: **BUY/ADD 2/day; TRIM/SELL exempt** with a 10/day circuit
  breaker as a malfunction brake.
- Holdings: **hard cap 10**; the ~7-holding goal lives in the mandate, not
  policy. Minimum initial position 5% (mandate-enforced).
- Theme concentration: **single primary theme ≤ 60%**, via the per-decision
  `primary_theme_id` field.
- Drawdown: **regime model only**, no drawdown triggers; compensating
  controls are the dashboard and evals (see `docs/risk_policy.md`, which
  records the accepted single-point-of-failure risk).
- X: curation human-only; expert-reader scrutiny doctrine; account scrutiny
  ledger (agent-computed, reduce-only); counter-evidence discipline;
  provider decision deferred to plan 008's research.
- `confirmed_outside_x` semantics redefined in `docs/source_policy.md`
  ("substantiated + survived expert scrutiny"); a field rename (e.g.
  `claims_substantiated`) is deferred — cheap to do later, not worth churn
  now.
